# 研发同步：原型变动说明

> 日期：2026-04-20  
> 适用对象：前端 / 后端 / 测试 / 产品研发协同  
> 说明：本文只描述当前原型层面的业务变化与展示口径，不描述代码实现细节。

---

## 一、转诊单详情页：流程进度图调整

### 1. 调整背景

原详情页顶部使用的是按角色和事件拆分的泳道图，强调“谁做了什么”。  
现调整为更适合详情页阅读的“转诊闭环信息”，强调“这张转诊单已经走到了哪一步”。

本次调整目标：

- 降低理解成本，避免用户把流程图看成角色职责图
- 让不同角色查看同一张转诊单时，顶部闭环信息保持一致
- 与底部“操作日志”分工清晰，避免重复展示

### 2. 调整后展示方式

顶部不再展示泳道式流程图，改为“关键事件闭环”展示。

统一展示 5 个关键事件节点：

1. 发起转诊
2. 转诊受理
3. 到院/接诊
4. 完成诊疗 / 接收完成
5. 闭环完成

不同转诊类型下，节点名称会做轻微适配，但结构保持一致：

- 普通上转 / 住院上转：按“发起 → 受理 → 到院/接诊 → 完成诊疗 → 闭环完成”展示
- 急诊 / 绿通上转：按“发起急诊/绿通 → 已通知并进入接诊准备 → 到院/接诊 → 急诊处置完成 → 闭环完成”展示
- 下转：按“发起下转 → 下转接收 → 患者转入基层 → 接收完成 → 闭环完成”展示

### 3. 状态含义

每个关键事件节点只展示四种状态：

- 已完成
- 进行中
- 待发生
- 已终止

说明：

- 顶部闭环是“摘要视图”，只表达主线进度
- 终止态适用于已拒绝、已撤销、已关闭等场景
- 当前进行中的节点会尽量显示可参考时间，避免整行空白

### 4. 与操作日志的分工

顶部闭环信息不再重复承载以下内容：

- 系统通知
- 短信发送记录
- 催办记录
- 改派记录
- 内部审核明细
- 每一条操作人动作
- 拒绝原因全文
- 补充上传类动作

这些内容继续放在详情页底部“操作日志”中查看。

新的展示原则是：

- 顶部闭环：回答“这张单走到哪了”
- 底部日志：回答“过程中具体发生了什么”

### 5. 研发理解要点

- 顶部流程图不再按角色泳道展示
- 同一张转诊单，不同角色看到的顶部闭环结构一致
- 闭环只保留关键业务节点，不承担明细审计职责
- 后续如补充真实事件字段，应优先服务于“关键事件摘要”，而不是再回到泳道图模式

## 二、CHG-42：上转单详情页信息区块按角色精准展示

### 1. 变更范围

本次 CHG-42 只针对 **上转单**：

- 普通上转
- 急诊上转
- 绿通上转

下转单同类问题后续另行处理，不在本次范围内。

### 2. 核心设计原则

详情页区块是否展示，不再按“能不能看”来判断，而按“这个信息对当前角色有没有业务作用”来判断。

统一原则：

- 有作用：完整展示
- 仅需知道流程是否推进：极简状态文字
- 无作用：不展示

这意味着：

- 不是所有已存在数据都要在每个角色详情页里展示
- 不能再用“仅供参考”之类免责语，把无关信息硬塞给不需要的角色

### 3. 角色口径调整

#### 3.1 基层医生 / 基层负责人

上转详情页中：

- 完整展示“到院接诊安排”
- 完整展示“转诊预约码”
- 可看到接诊时间、科室、楼层、诊室/床位、科室电话、接诊医生（如有）
- 增加面向基层医生的提示语：
  - 请告知患者按以下信息到院就诊
  - 实际接诊医生以到院现场安排为准

业务原因：

- 基层侧需要承担向患者转告就诊信息的职责
- 预约码和接诊安排对患者到院动作有直接指导价值

#### 3.2 县级经办医生 / 县级科主任

上转详情页中：

- 不再完整展示“到院接诊安排”
- 改为只展示一条极简状态信息
- 不展示“转诊预约码”
- 不展示“就诊须知”

极简状态信息分两种：

- 已有接诊安排：
  - 转诊中心已完成接诊安排，患者预计某时到某科室就诊
- 尚无接诊安排：
  - 转诊中心正在安排接诊资源，请等待

业务原因：

- 县级经办医生需要知道流程是否已进入下一环节
- 但他不负责向患者转告预约码，也不负责解释到院须知
- 详细安排信息放给县级医生反而容易造成“为什么不是我接诊”的理解偏差

#### 3.3 转诊中心 / 管理员

上转详情页中：

- 保持完整展示“到院接诊安排”
- 保持完整展示“转诊预约码”
- 保持患者短信预览入口
- 不展示“就诊须知”区块本体

业务原因：

- 转诊中心负责安排到院资源和短信通知，是这部分信息的真正执行角色

### 4. 区块级变化

#### 4.1 转诊预约码

调整后口径：

- 基层医生 / 基层负责人：显示
- 转诊中心 / 管理员：显示
- 县级医生 / 县级科主任：不显示
- 急诊 / 绿通上转：所有角色统一不显示

说明：

- 急诊和绿通本身不走预约码逻辑
- 非急诊场景下，预约码属于患者到院动作所需信息，不属于县级经办医生的工作信息

#### 4.2 到院接诊安排

调整后口径：

- 基层医生 / 基层负责人：完整展示
- 转诊中心 / 管理员：完整展示
- 县级医生 / 县级科主任：极简状态展示

“完整展示”包含的信息：

- 就诊时间
- 接诊科室
- 楼层 / 区域
- 诊室 / 床位
- 科室电话
- 接诊医生（如有）
- 住院分支时的病区 / 床位号 / 护士站信息

“极简状态展示”只表达：

- 是否已经完成安排
- 患者预计何时去哪个科室

#### 4.3 就诊须知

调整后口径：

- 所有系统内医生角色详情页中统一不展示

业务归属调整为：

- 仅作为患者短信末尾固定段落存在
- 不再作为系统内详情页区块存在

### 5. 操作日志口径补充

对于上转单：

- “操作日志”不再作为所有角色都能看到的详情页内容
- 主要保留给转诊中心 / 管理员查看

研发理解重点：

- 上转详情页强调“角色相关信息”
- 日志区强调“协同记录与审计记录”
- 二者分层，不再混合

### 6. F-02 原型展示调整

本次原型中，机构配置页新增了“患者须知模板”配置项，用于表达 CHG-42 的业务归属变化。

该配置项的业务含义：

- 属于机构级患者须知模板
- 用于接诊安排完成后的患者通知短信
- 支持替换模板中的占位符
- 当前原型仅做静态展示和短信预览拼接，不要求真实保存和真实生效链路

当前原型支持表达的占位符包括：

- 接诊科室
- 预约码
- 病区
- 床位号
- 护士站电话

### 7. 患者短信口径

短信预览中，现按 CHG-42 的业务要求体现：

- 上半段展示到院信息 / 接诊信息
- 下半段追加“就诊须知”模板内容

业务上要理解为：

- “就诊须知”不再属于医生详情页信息
- “就诊须知”属于患者触达信息

### 8. 本次原型不涉及的内容

以下内容本次不作为改动范围：

- 下转单详情页同类逻辑
- 状态机流转本身
- 按钮权限本身
- 预约码生成规则
- 接诊安排数据结构本身
- 真实短信服务
- 真实后端配置联动

## 三、给研发的落地理解建议

### 1. 不要把这次改动理解成“UI 调整”

这次变化本质上是 **详情页信息架构重构**：

- 哪些信息应该给谁看
- 哪些信息只保留流程感，不给执行细节
- 哪些信息不应该继续放在系统详情页里

### 2. 顶部闭环和详情信息区块要分开理解

- 顶部闭环：主线进度
- 详情区块：角色相关业务信息
- 底部日志：明细记录

三者各司其职，不能混用。

### 3. 原型阶段的关键验收点

研发联调时，建议优先按以下业务口径验收：

- 同一张上转单，基层和县级看到的详情区块不再相同
- 县级详情页不再出现预约码、完整接诊安排、就诊须知
- 基层详情页能完整看到患者到院所需信息
- 管理员仍保留完整接诊安排和短信预览
- 急诊/绿通上转不显示预约码
- 顶部闭环与底部日志不再表达同一层信息
